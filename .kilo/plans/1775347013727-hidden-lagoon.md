# Plano: Funções auxiliares para montagem de modlist no SDK

## Objetivo
Criar API helpers no SDK para facilitar a criação e composição de modlists, reduzindo boilerplate e melhorando a experiência do usuário final.

---

## Contexto Atual
Hoje o usuário precisa criar Shards manualmente, lidar com versões, dependências e composição diretamente. Não existem funções auxiliares para:
  - Agrupar mods
  - Aplicar configurações globais
  - Gerenciar ordem de sobreposição
  - Criar perfis
  - Filtrar e transformar listas de mods

---

## Propostas de API

### 📌 Proposta 1: `modlist()` - Construtor principal
```typescript
// Opção A - Fluent Builder
const myModpack = modlist()
  .name("My Skyrim Modpack")
  .version("1.0.0")
  .add(modA)
  .add(modB, { after: modA })
  .addAll([modC, modD, modE])
  .exclude(modF)
  .configure({ loadOrder: "auto" })
  .build();

// Opção B - Declarativo
const myModpack = modlist({
  name: "My Skyrim Modpack",
  version: "1.0.0",
  mods: [
    modA,
    { mod: modB, after: modA },
    ...commonMods,
  ],
  exclude: [modF],
  options: {
    resolveConflicts: true,
    autoLoadOrder: true
  }
});
```

### 📌 Proposta 2: `group()` - Agrupamento lógico
```typescript
const visualMods = group("Visual Overhauls", [
  enbPreset,
  weatherMod,
  texturePack,
  grassMod
]);

const gameplayMods = group("Gameplay", [
  combatOverhaul,
  perkOverhaul,
  survivalMod
]);

// Groups são compostáveis
const fullModpack = modlist()
  .add(visualMods)
  .add(gameplayMods)
  .build();
```

### 📌 Proposta 3: `profile()` - Perfis alternativos
```typescript
const base = modlist().add(commonMods);

const vanillaPlus = profile("Vanilla Plus", base)
  .add(lightMods)
  .remove(heavyOverhauls);

const modded = profile("Full Modded", base)
  .add(heavyOverhauls)
  .add(additionalContent);
```

### 📌 Proposta 4: Funções de manipulação
```typescript
// Ordenação
sortByPriority(mods, "desc");
orderBefore(modX, modY, modList);
orderAfter(modY, modX, modList);

// Filtros
onlyEnabled(mods);
excludeTags(mods, ["performance-heavy"]);
includeOnlyTags(mods, ["visual", "bugfix"]);

// Transformações
mapMods(mods, mod => ({ ...mod, version: "latest" }));
overrideProperty(mods, "postbuild", "patch.sh");
```

## ✅ Decisões Tomadas
1. **Estilo de API**: Apenas **Declarativo**, alinhado com a filosofia do Kintsugi. Nenhum builder fluente.
2. **Prioridade**: Primeiro implementar composição e estrutura base, depois helpers para manipulação de mods.
3. **`compose()`**: Manter existente, `modlist()` será uma camada superior que usa `compose()` internamente.
4. **Tags**: Adiar para versões futuras, não implementar nesta iteração.
5. **Funcionalidades avançadas**: Filtros, ordenação customizada, perfis serão implementados depois da base estar pronta.

## Estrutura Final da API

### `modlist()` - Função principal
```typescript
import { modlist } from "@btripoloni/kintsugi";

export default modlist({
  name: "Skyrim SE Modpack",
  version: "1.0.0",

  // Mods são colocados na ordem informada (último tem prioridade máxima)
  mods: [
    skyrimBaseGame,
    ussep,
    skse,
    enb,
    weatherMod,
    combatOverhaul
  ],

  // Opções
  options: {
    autoResolveDependencies: true,
    deduplicate: true
  }
});
```

### ✅ Comportamento Oficial de Dependências
Essa é a regra mais importante:

> **Quando você coloca `modB` e `modC` no `modlist()`, e ambos declaram `modA` como `dependencies` no seu Shard:**
> 1. ✅ `modA` é **automaticamente incluído** na composição
> 2. ✅ `modA` é **colocado ANTES** de `modB` e `modC` na ordem de layers
> 3. ✅ `modA` aparece **apenas uma vez** mesmo que seja dependência de 100 mods diferentes
> 4. ✅ A ordem final é garantida: `[ modA, modB, modC ]`

Esse comportamento **já existe hoje no `compose()`** e `modlist()` só vai formalizar e garantir que ele sempre funcione dessa forma. Nenhuma mágica nova, só estamos padronizando o comportamento que já deveria existir.

### Estrutura interna:
- `modlist()` retorna um Shard normal do tipo `composition`
- É 100% compatível com toda API existente
- Pode ser passado diretamente para `compose()`, usado como dependência, ou exportado como entrada principal do modlist
- Internamente `modlist()` vai chamar `resolveTransitiveLayers()` do SDK que já implementa essa lógica correta de topo ordenação e deduplicação

## Roadmap de Implementação

### Fase 1 (Obrigatória, primeiro):
1. ✅ `resolveTransitiveLayers()` já existe e implementa exatamente o comportamento correto (linha 4 `modpack.ts`)
   - Já faz ordenação topológica
   - Já deduplica shards por hash
   - Já detecta dependências circulares
   - Já respeita ordem de dependências

2. Adicionar interface `ModlistOptions` no `modpack.ts`
3. Implementar função `modlist()` que:
   ```typescript
   export async function modlist(options: ModlistOptions): Promise<Shard> {
       return compose({
           name: options.name,
           version: options.version,
           layers: options.mods
       });
   }
   ```
4. Exportar `modlist` no `index.ts` do SDK
5. Escrever testes unitários cobrindo:
   - Dependências transitiveis são automaticamente incluídas
   - Ordem correta: dependências sempre vem primeiro
   - Deduplicação funciona mesmo com múltiplas referências
   - 100% compatibilidade com Shard e compose() existente

Obs: O `modlist()` é literalmente só um alias melhor nomeado para `compose()`. Todo o trabalho duro já está implementado e funcionando corretamente.


### Fase 2 (Depois):
Funcionalidades adicionais que são opcionais na primeira versão mas serão adicionadas logo depois:

1. **Ordenação explícita**
   Você poderá sobrescrever a ordem de qualquer mod individualmente sem reordenar toda a lista:
   ```typescript
   modlist({
     mods: [
       baseMod,
       modA,
       modB,
       // modC sempre carrega DEPOIS de modB, mesmo que apareça antes na lista
       { mod: modC, after: modB },
       modD
     ]
   })
   ```

2. **Exclusão de mods**
   Útil quando você importa grupos prontos e quer remover apenas um item específico:
   ```typescript
   modlist({
     mods: [
       ...allBugfixes,
       ...visualMods
     ],
     // Remove este mod mesmo que ele esteja nos grupos importados
     exclude: [ brokenMod ]
   })
   ```

3. **Grupos lógicos**
   Para organizar mods em categorias sem alterar o comportamento final:
   ```typescript
   const bugfixes = group("Correções", [ ussep, engineFixes, crashFixes ]);
   const visuals = group("Visual", [ enb, weather, textures ]);

   modlist({ mods: [ bugfixes, visuals ] })
   ```

4. **Perfis**
   Cria variantes do mesmo modlist base:
   ```typescript
   const base = modlist({ mods: [ commonMods ] });

   // Perfil leve
   const light = profile(base, { add: [ lightMods ], remove: [ heavyMods ] });

   // Perfil completo
   const full = profile(base, { add: [ allMods ] });
   ```

> 💡 Importante: Todas essas funcionalidades são **açúcar sintático**. Nenhuma delas altera o resultado final do build ou quebra compatibilidade. Elas só facilitam para você escrever o modlist. Todas elas no final geram exatamente o mesmo array de Shards que você escreveria manualmente hoje.

## 🆕 Proposta: API para criação de Shards

Atualmente o usuário tem que escrever objetos Shard manualmente. Vamos criar helpers tipados para cada tipo de source.

### Proposta de API:
```typescript
import { url, local, json, run, vase } from "@btripoloni/kintsugi";

// URL Source
const skse = url({
  name: "skse",
  version: "2.2.6",
  url: "https://skse.silverlock.org/skse64_2_02_06.7z",
  sha256: "abc123def...",
  unpack: true,
});

// Local Source
const myMod = local({
  name: "my-custom-mod",
  version: "1.0.0",
  path: "./mods/my-mod",
});

// JSON File
const config = json({
  name: "mod-config",
  version: "1.0.0",
  path: "SKSE/Plugins/Config.json",
  content: { enabled: true, value: 42 },
});

// Execução
const profile = run({
  name: "launch-profile",
  version: "1.0.0",
  entrypoint: "skse64_loader.exe",
  args: ["--skip-intro"],
});
```

### Vantagens:
✅ Tipagem 100% correta, erros no momento da escrita
✅ Não precisa lembrar dos nomes dos campos de cada source
✅ Validações automáticas
✅ Menos boilerplate
✅ Consistência entre todos os shards

✅ Todos os helpers retornam um Shard normal que pode ser passado diretamente para `modlist()`, usado como dependência, ou em qualquer lugar que aceite Shard.
